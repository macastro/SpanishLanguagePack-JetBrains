Informa sobre clases, métodos o campos no utilizados o inaccesibles desde los puntos de entrada.

Los puntos de entrada pueden ser métodos main, pruebas, clases fuera del alcance especificado, clases accesibles desde module-info.java, etc. Los puntos de entrada personalizados se pueden configurar mediante patrones de nombres o anotaciones.

Ejemplo:


  public class Department {
    private Organization myOrganization;
  }

En este ejemplo, Department hace referencia explícita a Organization, pero si la clase Department no se utiliza, la inspección informará sobre ambas clases.

La inspección también informa sobre parámetros cuyos métodos, todas las implementaciones de métodos y anulaciones no se utilizan, así como variables locales declaradas pero no utilizadas.

Nota: Es posible que algunos miembros no utilizados no se informen durante el resaltado de código en el editor. Por razones de rendimiento, los miembros no privados solo se verifican cuando sus nombres aparecen raramente en el proyecto. Para ver todos los resultados, ejecute la inspección seleccionando Código | Inspeccionar código o Código | Analizar código | Ejecutar inspección por nombre desde el menú principal.

Los miembros a informar se pueden configurar usando la configuración de visibilidad a continuación. Por ejemplo, configurar para informar métodos private solo indica que se informarán los métodos public de clases internas private, mientras que los métodos protected de clases de nivel superior serán ignorados.

Use la pestaña Puntos de entrada para configurar los puntos de entrada a considerar durante la ejecución de la inspección.

Los puntos de entrada se pueden agregar manualmente una vez que los resultados de la inspección estén listos.

Si su código utiliza frameworks no soportados, puede elegir entre las siguientes opciones:

  • Si el framework depende de anotaciones, use el botón Anotaciones... para configurar las anotaciones del framework.
  • Si el framework no depende de anotaciones, puede intentar configurar patrones de nombres de clase que el framework debería tener.

De esta manera, el código anotado accesible dentro del framework se considerará como utilizado.
