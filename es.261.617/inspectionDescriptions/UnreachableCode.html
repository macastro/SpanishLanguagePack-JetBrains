Reporta código que nunca se alcanza según el análisis de flujo de datos. Puede ser el resultado de una condición anterior que siempre es true o siempre es false, un cuerpo de bucle inalcanzable o una parte catch. Usualmente (aunque no siempre) el código inalcanzable es el resultado de advertencias previas, por lo que revise las advertencias de las inspecciones que forman "Problemas de nullabilidad y flujo de datos", "Valores constantes" o "Operaciones redundantes en contenedores vacíos" para entender mejor la causa.

Ejemplo:


  void finishApplication() {
    System.exit(0);
    System.out.println("Application is terminated"); // código inalcanzable
  }

Tenga en cuenta que esta inspección depende de la inferencia de contratos de métodos. En particular, si llama a un método static o final que siempre lanza una excepción, se inferirá el contrato "siempre falla", y el código después de la llamada al método será considerado inalcanzable. Por ejemplo: 


  void run() {
    performAction();
    System.out.println("Action is performed"); // código inalcanzable
  }
  static void performAction() {
    throw new AssertionError();
  }

Esto puede causar falsos positivos si usa cualquier tipo de postprocesamiento de código, por ejemplo, si un procesador de anotaciones reemplaza posteriormente el cuerpo del método con contenido útil. Para evitar advertencias de falsos positivos, use la anotación explícita @org.jetbrains.annotations.Contract del paquete org.jetbrains:annotations para suprimir la inferencia automática de contratos: 


  void run() {
    performAction();
    System.out.println("Action is performed"); // ya no hay advertencia
  }
  @Contract("-> _") // la implementación será reemplazada
  static void performAction() {
    throw new AssertionError();
  }
