Informa sobre código al que nunca se llega basándose en el análisis de flujo de datos. Puede ser el resultado de una condición previa que siempre es verdadera o siempre falsa, un cuerpo de bucle inalcanzable o una sección catch. A menudo (aunque no siempre), el código inalcanzable es consecuencia de advertencias previas, por lo que conviene revisar las advertencias de las inspecciones de "Problemas de nulabilidad y flujo de datos", "Valores constantes" u "Operaciones redundantes en contenedores vacíos" para comprender mejor la causa.

Ejemplo:


  void finishApplication() {
    System.exit(0);
    System.out.println("Application is terminated"); // Código inalcanzable
  }

Tenga en cuenta que esta inspección depende de la inferencia de contratos de métodos. En particular, si llama a un método static o final que siempre lanza una excepción, se inferirá un contrato de "siempre falla" y el código posterior a la llamada al método se considerará inalcanzable. Por ejemplo: 


  void run() {
    performAction();
    System.out.println("Action is performed"); // Código inalcanzable
  }
  
  static void performAction() {
    throw new AssertionError();
  }

Esto puede provocar falsos positivos si se utiliza algún tipo de postprocesamiento de código; por ejemplo, si un procesador de anotaciones reemplaza posteriormente el cuerpo del método con contenido útil. Para evitar advertencias de falsos positivos, utilice la anotación explícita @org.jetbrains.annotations.Contract del paquete org.jetbrains:annotations para suprimir la inferencia automática de contratos: 


  void run() {
    performAction();
    System.out.println("Action is performed"); // Ya no hay advertencia
  }

  @Contract("-> _") // La implementación será reemplazada
  static void performAction() {
    throw new AssertionError();
  }
