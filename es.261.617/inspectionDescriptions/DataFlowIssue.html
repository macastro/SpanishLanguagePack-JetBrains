Según el análisis de flujo de datos, reporta código que siempre viola las convenciones de nulabilidad, puede lanzar excepciones o simplemente estructuras de código redundantes.

Ejemplo:

if (array.length < index) {
  System.out.println(array[index]);
} // El índice del array siempre está fuera de los límites
if (str == null) System.out.println("str is null");
System.out.println(str.trim());
// La última declaración puede lanzar NPE
@NotNull
Integer square(@Nullable Integer input) {
    // Viola el contrato del método
    return input == null ? null : input * input;
}

El comportamiento de la inspección puede ser controlado por muchas anotaciones, como las anotaciones de nulabilidad, la anotación @Contract, la anotación @Range, etc.

Configurar la inspección:

  • Usa la opción Sugerir anotación @Nullable para métodos/campos/parámetros que pueden ser null para emitir advertencias cuando: valores null se pasan como argumentos a métodos con parámetros no anotados, se almacenan en campos no anotados, o se devuelven desde métodos no anotados. En este caso, la inspección sugerirá propagar la anotación @Nullable. También puedes configurar las anotaciones de nulabilidad usando el botón Configurar anotaciones.
  • Usa la opción Tratar miembros y parámetros no anotados como @Nullable para asumir que los miembros no anotados pueden ser null, por lo tanto no deben usarse en contextos no null.
  • Usa la opción Reportar parámetros requeridos no null con uso de argumentos literales null para reportar parámetros de método que no pueden ser null (por ejemplo, se desreferencian inmediatamente en el cuerpo del método), pero tienen puntos de llamada que pasan el literal null.
  • Usa la opción Reportar métodos nullable que siempre devuelven valores no null para reportar métodos anotados como @Nullable pero que siempre devuelven valores no null. En este caso, se sugiere cambiar la anotación a @NotNull.
  • Usa la opción Ignorar declaraciones assert para controlar cómo la inspección maneja las declaraciones assert. Por defecto, esta opción está deshabilitada, lo que significa que las aserciones se consideran ejecutadas (modo -ea). Si se habilita la opción, las aserciones serán completamente ignoradas (modo -da).
  • Usa la opción Reportar problemas que ocurren solo en algunas rutas de código para controlar si se reportan problemas que ocurren solo en algunas rutas de código. Si esta opción está deshabilitada, no se reportarán advertencias como exception is possible. La inspección solo reportará advertencias como exception will definitely occur. Este modo puede reducir significativamente el número de falsos positivos, especialmente si el código no está consistentemente anotado con anotaciones de nulabilidad y contrato. Por eso es útil para encontrar los problemas más importantes en bases de código antiguas.

Antes de la versión IntelliJ IDEA 2022.3, esta inspección era parte de la inspección "Condiciones constantes y excepciones". Ahora, se ha dividido en dos inspecciones: "Valores constantes" y "Problemas de nulabilidad y flujo de datos".
