Informa de infracciones de la Ley de Demeter.

La Ley de Demeter no es realmente una ley, sino que especifica una directriz de estilo: nunca llames a un método en un objeto recibido de otra llamada. El código que sigue esta directriz es más fácil de mantener, adaptar y refactorizar, tiene menos acoplamiento entre métodos, menos duplicación y mejor ocultación de información. Por otro lado, es posible que necesites escribir muchos métodos wrapper para cumplir con esta directriz.

Ejemplo:


  boolean pay(Customer c, Invoice invoice) {
    int dollars = c.getWallet().contents; // violación
    if (dollars >= invoice.getAmount()) {
      Wallet w = c.getWallet();
      w.subtract(invoice.getAmount()); // violación
      return true;
    }
    return false;
  }
El ejemplo anterior podría implementarse mejor como un método payInvoice(Invoice invoice) en Customer.

Utiliza la opción Ignorar llamadas a métodos de biblioteca y acceso a campos de biblioteca para ignorar las infracciones de la Ley de Demeter que no se pueden corregir sin cambiar una biblioteca.
