Reporta clases, métodos o campos que no se usan o que son inalcanzables desde los puntos de entrada.

Un punto de entrada puede ser un método principal, pruebas, clases externas al ámbito especificado, clases accesibles desde module-info.java, y demás. Es posible configurar puntos de entrada personalizados mediante el uso de patrones de nombre o anotaciones.

Ejemplo:


  public class Department {
    private Organization myOrganization;
  }

En este ejemplo, Department explícitamente hace referencia a Organization pero si la clase Department en sí es inutilizada, entonces la inspección reportará ambas clases.

La inspección también reporta parámetros que no son usados por sus métodos y todas las implementaciones y anulaciones de métodos, así como variables locales que son declaradas pero no usadas.

Nota: Algunos miembros inutilizados pueden no ser reportados durante el resaltado de código en el editor. Por razones de rendimiento, solo se verifica un miembro no privado cuando rara vez aparece su nombre en el proyecto. Para ver todos los resultados, ejecuta la inspección seleccionando Código | Inspeccionar código o Código | Analizar código | Ejecutar inspección por nombre en el menú principal.

Utiliza las siguientes configuraciones de visibilidad para configurar los miembros a ser reportados. Por ejemplo, configurar para reportar solo métodos private significa que los métodos public de la clase interna private serán reportados pero los métodos protected de la clase de nivel superior serán ignorados.

Usa la pestaña puntos de entrada para configurar los puntos de entrada a ser considerados durante la ejecución de la inspección.

Puedes añadir puntos de entrada manualmente cuando los resultados de la inspección estén listos.

Si tu código utiliza frameworks no soportados, hay varias opciones:

  • Si el framework se basa en anotaciones, usa el botón Anotaciones... para configurar las anotaciones del framework.
  • Si el framework no se basa en anotaciones, intenta configurar patrones de nombre de clase que el framework espere.

De esta forma, el código anotado accesible por los componentes internos del framework será tratado como usado.
